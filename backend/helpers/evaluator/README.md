# Evaluator Agent

## Overview

The **Evaluator Agent** is responsible for verifying code generated by the **LLM (Large Language Model)** by comparing it against an existing domain-specific repository. The goal is to analyze key components of the generated code and determine whether they already exist in the repository or are newly introduced by the model.

The Evaluator Agent ensures that the generated code maintains consistency with the existing codebase, providing users with insights into how much of the generated code is leveraging existing components.
It achieves this by analyzing key components in the newly generated code and comparing them against a domain-specific repository that the model has been trained on.


## Workflow

Primary Agent  → Evaluator Agent → Verification Result
     ↑                           ↑
     |                          |
Code Repository  →  Code Analysis DB/Index


## Functionality

When the LLM generates a new response containing code, the **Evaluator Agent** analyzes its key components and determines whether they already exist in the domain-specific repository. This helps users understand whether the LLM is leveraging existing code from the original repository when generating suggestions.

For example, if the generated code is in **Go**, the Evaluator will identify **packages, interfaces, structs, functions, and methods** and check whether they exist in the repository.

### Current Limitations

1. **Go-Specific Analysis**:  
   - Currently, we only have an analyzer for **Go**.  
   - Other languages are not yet supported.

2. **Regex-Based Parsing**:  
   - The Go analyzer is **regex-based**, which limits its ability to parse complex code snippets accurately.  
   - This may lead to **missed** or **incorrectly parsed** key elements.

3. **Handling New Code as Non-Existent**:  
   - The Evaluator treats **newly generated code** as **NOT_EXIST** if it does not appear in the original repository.  
   - This may incorrectly lower the overall matching score.  
   - Example:  
     - If the LLM generates a new function `helper_func()` that is not in the original repository, it will be marked as **NOT_EXIST**.  
     - However, the function may be valid and expected, making this a **potential misclassification**.

## Enabling the Evaluator Agent

To enable the **Evaluator Agent** for a specific project, you must add the `repo_internal_location` key to the corresponding **model/adapter** in the database. The value should be the **path** on the backend hosting machine where the fine-tuned model's repository is stored.

Example:
```json
{
  "repo_internal_location": "/tmp/oadp/oadp-e2e-qe"
}

## Execution Flow

Currently, the Evaluator Agent runs on-the-fly during each evaluation:

1. Analyzes the entire repository (each time a new evaluation is triggered).
2. Extracts and analyzes the new LLM-generated code.
3. Runs the matching algorithm to determine similarities and differences.

## Future Optimizations

To improve performance, we may integrate the evaluator at application startup, storing analyzed repository data in a dedicated database.

### Example of Evaluator Integration:

```python
evaluator = EvaluatorAgent(repo_path)
evaluator.analyzer.analyze_repository(repo_path)
```

By caching repository analysis results, we can significantly reduce processing time during evaluations.

# **Conclusion**  

The **Evaluator Agent** plays a crucial role in verifying LLM-generated code against an existing codebase. While current limitations exist, future enhancements, such as **precomputed analysis storage** and **improved parsing techniques**, will enhance its accuracy and efficiency.  

## **Future Enhancements Considerations**  

- **Improve parsing capabilities** beyond regex-based analysis.  
- **Extend support** to additional programming languages.  
- **Optimize the matching algorithm** to handle newly generated functions more intelligently.
- **Optimize evaluation speed** by caching repository analysis results in a database to speed up evaluations
